# 3.5 Migration Plan — Expose serve mode over MCP (in addition to HTTP/S REST)

> Status: **plan only** — no code changes are made until implementation is
> explicitly requested. Follows the repo migration-plan convention
> (cf. `migration/3.2_MIGRATION_PLAN_EWS.md`).

## Context

`gomailtest serve` (`internal/serve/`) today runs an HTTP REST server that sends
email via `POST /smtp/sendmail` and `POST /msgraph/sendmail`, loading credentials
from env vars at startup and reusing the core send logic
(`smtp.SendMail`, `msgraph.SendEmail`). The request bodies carry only message
content; a shared `X-API-Key` protects every non-health route.

We want the **same sendmail capabilities reachable over MCP** (Model Context
Protocol), so AI clients (Claude Desktop/Code, IDEs) and networked MCP callers can
drive it — without duplicating the send/validation logic that already backs the
REST handlers.

**Decision (user-confirmed): expose MCP over BOTH transports** from a single
`mcp.Server`:
- **Streamable HTTP** mounted at `/mcp` on the existing serve HTTP server, behind
  the same `X-API-Key` middleware (the "on HTTP server but also MCP" reading).
- **stdio** via a new `serve --mcp-stdio` mode, for local AI clients that launch
  the binary as a subprocess.

## Library

Official Go SDK, **`github.com/modelcontextprotocol/go-sdk/mcp`** (stable v1.x).
Verified API:
- `mcp.NewServer(&mcp.Implementation{Name, Version}, nil) *mcp.Server`
- `mcp.AddTool(server, &mcp.Tool{Name, Description}, handler)` where the handler is
  `func(ctx, *mcp.CallToolRequest, ArgsStruct) (*mcp.CallToolResult, any, error)`;
  the SDK derives the input JSON schema from the `ArgsStruct` json tags.
- stdio: `server.Run(ctx, &mcp.StdioTransport{})` (blocks until closed).
- HTTP: `mcp.NewStreamableHTTPHandler(func(*http.Request) *mcp.Server, *mcp.StreamableHTTPOptions) *mcp.StreamableHTTPHandler`
  — implements `http.Handler`, mountable on the existing `http.ServeMux`.

Add via `go get github.com/modelcontextprotocol/go-sdk@latest`; confirm exact
signatures against the pinned version after `go mod tidy`.

## Core refactor (the reuse boundary)

The single most important change: **extract everything after JSON-decode** in the
two HTTP handlers into transport-agnostic methods, so HTTP and MCP share one code
path. Make the boundary the *parsed request struct*, not the HTTP handler.

In a new `internal/serve/send.go` (or within existing files):

```go
// Classified error so each transport maps failures to its own status model.
type sendError struct { kind sendErrKind; err error } // kind: badRequest | unavailable | internal

func (s *Server) sendSMTP(ctx context.Context, req smtpSendRequest, csv logger.Logger) *sendError
func (s *Server) sendMsgraph(ctx context.Context, req msgraphSendRequest, csv logger.Logger) *sendError
```

- Move the validation + config-overlay + `smtp.SendMail` / `msgraph.SendEmail`
  calls (currently in `smtp_handler.go` / `msgraph_handler.go`) into these methods.
- Reuse the existing `smtpSendRequest` / `msgraphSendRequest` structs unchanged —
  they double as the MCP tool arg structs (json tags drive both).
- The CSV logger is passed **in** (not created inside) so each transport controls
  logging destination — see Stdout hygiene below.

Then:
- **HTTP handlers** become thin: decode JSON → build CSV logger as today → call
  core → map `sendError.kind` to the existing **400 / 503 / 500** and the
  `apiResponse` envelope. Keep this mapping identical so `server_test.go` passes
  **unchanged** — that is the regression guard for the refactor.
- **MCP tool handlers** (new `internal/serve/mcp.go`): args → call core → on
  `sendError` return `&mcp.CallToolResult{IsError: true, Content: ...message}`, else
  a success result. No CSV stdout banner in stdio mode.

## MCP server + tools (`internal/serve/mcp.go`)

`func (s *Server) newMCPServer() *mcp.Server` that registers, mirroring today's
serve endpoints (scope guard: **only sendmail**, not the other CLI test actions):

- `smtp_sendmail`  — args = `smtpSendRequest` (to, cc, bcc, from, subject, body, priority)
- `msgraph_sendmail` — args = `msgraphSendRequest` (to, cc, bcc, subject, body, bodyHTML, priority).
  **Keep the no-attachments restriction** (same path-traversal reason as REST).
- `list_backends` — mirror of `GET /` summary, so a client can discover which
  backends are configured (smtp/msgraph availability).

Unconfigured backend → always register the tool, return an error result on call
(the shared core already yields this, matching the REST `503`). Simpler than
conditional registration and keeps parity.

## Command surface & wiring (`internal/serve/cmd.go`)

Keep credential/config loading in `NewCmd()` exactly as-is (SMTP*/MSGRAPH* env +
`--config` sections), then branch on a new flag:

- Add flag `--mcp-stdio` (bool, env `SERVE_MCP_STDIO`).
- Add flag `--mcp` (bool, default **true**, env `SERVE_MCP`) to toggle mounting
  `/mcp` on the HTTP server.
- `RunE`: build the `*Server` as today, then:
  - if `--mcp-stdio`: call a new `Server.RunMCPStdio(ctx)` → `mcpSrv.Run(ctx, &mcp.StdioTransport{})`. Do **not** start the HTTP listener in this mode.
  - else: `Server.Run(ctx)` (HTTP), and when `--mcp` is set, mount the streamable
    HTTP handler.

In `Server.Run` (`internal/serve/server.go`), after building `mux`:
```go
if s.config.EnableMCP {
    mcpHandler := mcp.NewStreamableHTTPHandler(func(*http.Request) *mcp.Server { return s.newMCPServer() }, nil)
    mux.Handle("/mcp", mcpHandler)   // also "/mcp/" if the SDK expects a subtree
}
```
`apiKeyMiddleware` already wraps the whole mux, so `/mcp` sits behind `X-API-Key`
automatically — MCP must not become an auth bypass. Add `/mcp` to the `GET /`
summary and its availability. Extend `Config` (`internal/serve/config.go`) with
`EnableMCP bool` (and any stdio flag state needed).

## Stdout hygiene (stdio mode only) — as built

MCP-over-stdio uses **stdout** as the JSON-RPC channel; any stray write corrupts it.
The contamination sources are broader than just the CSV banner:
- slog → **stderr** already (`internal/common/logger/slog.go:29`) — safe.
- CSV logger prints `Logging to: <path>` to **stdout** (`csv.go:63`).
- **The reused send functions print progress to stdout**: `smtp.SendMail`
  (`fmt.Printf("Sending test email…")`, `"✓ Connected"`, …) and
  `msgraph.SendEmail`/`logVerbose` write human-readable output via `fmt.Printf`.

Solution (implemented): in `serve --mcp-stdio`, at the **top of `RunE`** —
*before* any logging is initialised — preserve the real stdout and repoint the
process's `os.Stdout` to `os.Stderr`:
```go
if mcpStdio {
    mcpStdout = nopWriteCloser{os.Stdout} // preserved real stdout for the transport
    os.Stdout = os.Stderr                 // all fmt.Printf / banners now go to stderr
}
```
Ordering matters: the redirect precedes `InitLoggers`, so the CSV banner (and
everything else) lands on stderr. In stdio mode the CSV file logger is not created
at all — `logger.SetupLogger` provides slog only — and the per-call CSV logger is
passed as `nil` (the send functions tolerate nil).

The MCP JSON-RPC channel then runs over the **preserved** stdout via
`mcp.IOTransport{Reader: os.Stdin, Writer: nopWriteCloser{realStdout}}` (chosen
over `StdioTransport`, which captures `os.Stdout` at Connect time). The writer is
wrapped so transport teardown does not close the real fd. HTTP and HTTP-mounted
MCP are unaffected (their stdout is not a protocol channel) and keep the CSV file
logger. No change needed to `csv.go` or the send functions.
